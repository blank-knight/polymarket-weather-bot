# Polymarket 天气交易 Bot — 程序设计文档

> 版本: v0.1 | 日期: 2026-04-05 | 状态: 草案

---

## 1. 项目目标

构建一个自动化交易 Bot，利用专业天气预报模型（GFS、ECMWF 等）与 Polymarket 天气预测市场之间的**信息时间差和定价偏差**，自动发现并执行高概率盈利交易。

**核心赚钱逻辑：**
```
天气模型每6小时更新 → Bot 秒级获取 → Polymarket 价格滞后 → 买入低估仓位 → 市场修正 → 获利
```

## 2. 目标市场

### 2.1 Polymarket 天气市场
- **市场规模**: 244个活跃市场，总交易量 $16.5M+
- **单个市场**: 日交易量 $300K-$400K
- **结算方式**: 次日结算（每日最高温市场）
- **市场类型**: "纽约4月6日最高温会在55-60°F之间吗？" → YES/NO 二元期权

### 2.2 市场结构
```
Event: "New York City High Temperature - April 6, 2026"
  ├── Market A: "Will high be 45-50°F?" → YES: $0.05 / NO: $0.95
  ├── Market B: "Will high be 50-55°F?" → YES: $0.15 / NO: $0.85
  ├── Market C: "Will high be 55-60°F?" → YES: $0.60 / NO: $0.40  ← 最可能
  ├── Market D: "Will high be 60-65°F?" → YES: $0.18 / NO: $0.82
  └── Market E: "Will high be 65-70°F?" → YES: $0.02 / NO: $0.98
```

### 3.2 五大核心模块

#### 模块 1: 天气数据引擎 (Weather Data Engine)
**职责**: 获取、融合、存储多源天气预报数据

- **数据源**:
  - **Open-Meteo API**（免费，聚合 GFS + ECMWF + ICON 等多模型）
    - GFS Ensemble API: 31成员集合预报，预报最长35天
    - ECMWF IFS: 欧洲模型，准确率最高
    - 集合预报 API: `/v1/ensemble` 端点
  - **NOAA NCEI 历史数据**: 10年历史温度基准概率
- **更新频率**: 每6小时（跟随 GFS/ECMWF 更新周期）
- **输出**: 针对目标城市的概率分布（正态CDF），包含:
  - 加权共识温度（ECMWF 0.35, GFS 0.25, UKMO 0.20, NWS 0.20）
  - 标准差（随预报时长动态调整: 6h→0.8°C, 10d+→5.5°C）
  - 离群值自动降权（0.5x）
  - 贝叶斯融合历史基准概率

#### 模块 2: 市场监控引擎 (Market Monitor Engine)
**职责**: 实时监控 Polymarket 天气市场价格和流动性

- **Polymarket API 架构**:
  - **Gamma API**: 市场元数据、发现、搜索
  - **CLOB API**: 中心限价订单簿，交易操作
  - **WebSocket**: `wss://ws-subscriptions-clob.polymarket.com/ws/market` 实时价格推送
- **功能**:
  - 自动发现新的天气市场（按城市、日期过滤）
  - 实时追踪所有天气 token 的 bid/ask 价格
  - 计算隐含概率分布
  - 监控市场流动性深度
- **目标城市**: 纽约、伦敦、巴黎、芝加哥、洛杉矶、东京等主要城市

#### 模块 3: 决策引擎 (Decision Engine)
**职责**: 比较天气预报概率与市场价格，发现定价偏差并做出交易决策

- **信号生成**:
  ```
  模型概率 P_model (from 模块1)
  市场价格 P_market (from 模块2)
  Edge = P_model - P_market

  当 |Edge| > Edge阈值 时，生成交易信号
  ```
- **仓位 sizing**: Kelly Criterion（凯利公式）
  - 使用 Quarter-Kelly（保守）: 最大单笔 15% bankroll
  - `f* = (p * b - q) / b`，其中 p=模型概率，b=赔率，q=1-p
  - 动态调整: 距结算越近，confidence 越高，可适当加仓
- **信号校准**: 基于历史交易结果，校准模型偏差
- **决策流程**:
  1. 获取最新天气模型概率分布
  2. 获取 Polymarket 各温度区间市场价格
  3. 计算每个区间的 Edge
  4. 过滤: Edge > 阈值 AND 流动性充足
  5. Kelly sizing 计算最优仓位
  6. 生成交易指令（方向、数量、限价）

#### 模块 4: 交易执行引擎 (Execution Engine)
**职责**: 通过 Polymarket CLOB API 执行交易

- **技术栈**:
  - `py_clob_client`: Polymarket 官方 Python CLOB 客户端
  - Polygon 链上交易（USDC 支付）
- **功能**:
  - API 密钥管理（derive-api-key 认证流程）
  - USDC/CTF token 授权管理
  - 限价单提交（GTC/FOK/IOC）
  - 仓位管理（持仓查询、平仓）
  - WebSocket 监控订单状态
- **执行策略**:
  - 新模型发布后立即抢跑（6小时周期窗口期）
  - 限价单挂单，避免吃单（节省 spread 成本）
  - 分批建仓，避免价格冲击

#### 模块 5: 风控模块 (Risk Management)
**职责**: 保护资金安全，防止重大亏损

- **规则**:
  - 单市场最大仓位: bankroll 的 15%（Quarter-Kelly 上限）
  - 总敞口上限: bankroll 的 50%
  - 单日最大亏损: bankroll 的 10%
  - 止损: 单笔亏损超过 30% 自动平仓
  - 结算前自动平仓时间: 结算前 2 小时
- **异常处理**:
  - API 断连 → 暂停新开仓，监控现有仓位
  - 价格异常波动 → 触发保护性平仓
  - 模型数据缺失 → 不交易

## 7. 核心时间线

```
T+0h: GFS/ECMWF 新模型发布
T+1s: Bot 获取新数据，计算概率分布
T+5s: 与 Polymarket 价格比较，计算 Edge
T+10s: Kelly sizing，生成交易信号
T+15s: 风控检查通过
T+20s: 提交限价单
T+1min~1h: 市场逐渐反应，价格修正
T+6h: 下一次模型更新，循环继续
```

**关键**: 竞争对手也是 bot，所以速度很重要，但天气市场不像加密市场那样毫秒级竞争，**6小时窗口期**给了足够的反应时间。

## 9. 风险与限制

1. **模型不完美**: 天气预报本身有误差，尤其 >7 天预测
2. **市场流动性**: 某些小城市市场流动性不足
3. **竞争加剧**: 越来越多 bot 进入，Edge 可能在缩小
4. **链上风险**: Polygon 网络拥堵、Gas 费波动
5. **API 限制**: Polymarket 可能有频率限制
6. **资金安全**: 私钥管理、合约风险
7. **监管风险**: 预测市场监管政策变化
